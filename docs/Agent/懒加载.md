# 懒加载注册器 (LazyAgentRegistry)

**源文件**: `agents/lazy_agent_registry.py`

## 概述

`LazyAgentRegistry` 是 CEOClaw **插件化智能体加载**的核心——它把 `.claude/skills/` 目录当作插件池,**按需**扫描、动态导入、延迟实例化。对外暴露类 dict 接口(`registry["research"]`),让上层(IntentionAgent / OrchestrationAgent)像查表一样拿到 Agent 实例,而不用关心底层模块在哪、何时加载。

核心理念是 **Progressive Disclosure**:启动时只"登记"哪些 skill 存在,真正被调度的那一个才会 `importlib` 加载 Python 模块并实例化。未被用户触发的 skill 永远不进内存。

## 加载流程

```
启动阶段 (__init__)
  └─ _discover_skills()
        └─ 扫描 .claude/skills/*/script/agent.py
        └─ 记录到 _skill_map: {skill_name: script_path}
        └─ 不加载 Python 模块

首次调用 registry["research"]
  └─ 查 cache — miss
  └─ _resolve_agent_name() — 解析别名
  └─ importlib.util.spec_from_file_location() — 动态加载模块
  └─ inspect 找到 AgentBase 子类
  └─ 按签名注入 model / memory_manager
  └─ 实例化 → 存入 cache

第二次调用 registry["research"]
  └─ 查 cache — hit,直接返回
```

- 启动阶段只做**目录扫描**,没有 `import`——即使某个 skill 的 `agent.py` 有语法错误,主进程也不会挂。
- 首次调用才 `exec_module`,错误就近抛出,调用方能明确知道是哪个 skill 出的问题。
- 实例化一次后常驻 `cache`,后续免开销。

## 类: `LazyAgentRegistry`

### `__init__(self, model, cache: Dict, memory_manager=None)`

| 参数 | 类型 | 说明 |
|---|---|---|
| `model` | LLM 模型实例 | 所有 Agent 共享同一个 model,避免重复建连 |
| `cache` | `Dict` | **外部传入**的字典,用于缓存已实例化的 Agent。调用方可以在外部读取/清空 |
| `memory_manager` | `MemoryManager` \| `None` | 可选;只有当 Agent 构造函数签名里有 `memory_manager` 参数时才注入 |

**内部状态**:
- `skills_root: Path` — 固定为 `.claude/skills`
- `_skill_map: Dict[str, Path]` — 发现的 skill 名 → `agent.py` 路径
- `_legacy_mapping: Dict[str, str]` — 旧命名(如 `research`)到新目录名的兼容映射

> `cache` 由外部传入而非内部创建,是为了让 CLI 能直接遍历"已加载 agent 列表"做状态展示/健康检查,同时也方便单元测试注入预构造实例。

---

### `_discover_skills() -> None`

扫描 `self.skills_root` 下所有一级目录,寻找 `<skill>/script/agent.py`,命中则登记到 `_skill_map`。

- **不 import 模块**,仅记录路径。
- 目录不存在时打 warning 不抛异常——允许在没有 skills 的环境下运行(CI 或裸测试)。
- `script/agent.py` 不存在的 skill 目录被**静默跳过**(可能只有 SKILL.md,没实现 agent)。

### `_resolve_agent_name(agent_name: str) -> Optional[str]`

两步解析:
1. `agent_name` 直接是 `_skill_map` 的 key → 返回
2. `agent_name` 在 `_legacy_mapping` 里,且映射目标存在 → 返回映射目标
3. 都没有 → `None`

存在这一层是为了**命名解耦**:调用方可以传 `"research"`(业务名),实际目录可以改成 `"market_research"` 而不破坏上层代码。

### `__getitem__(agent_name) -> AgentBase`

核心懒加载逻辑,按顺序执行:

1. **缓存命中** → 直接返回
2. **名称解析** → 失败抛 `KeyError`
3. **动态导入**:
   ```python
   spec = importlib.util.spec_from_file_location(f"skills.{skill_name}.agent", script_path)
   module = importlib.util.module_from_spec(spec)
   sys.modules[module_name] = module  # 关键:注册到 sys.modules 避免 pickle/re-import 问题
   spec.loader.exec_module(module)
   ```
4. **补 sys.path**:把项目根目录插入 `sys.path`,让 skill 内部能 `from utils import ...`、`from config import ...`。
5. **找 Agent 类**:`inspect.getmembers` 遍历模块成员,挑第一个 `AgentBase` 的非自身子类。
6. **按签名注入依赖**:
   - `name`、`model` 必传
   - 若构造函数有 `memory_manager` 参数,注入;否则不传。避免强制所有 skill 都依赖记忆系统。
7. **实例化 + 写入 `cache`**,返回实例。

失败时打印红色错误 + traceback 再 raise,便于 CLI 排查。

### `__contains__(agent_name) -> bool`

`"research" in registry` 的支撑。已缓存或能解析到 skill 都算存在——**不触发加载**。

### `get(agent_name, default=None)`

像 `dict.get` 一样,`KeyError` 时返回默认值。**会触发加载**(内部调 `__getitem__`),其他异常仍会抛出——这一点要留意:它不是"静默 fallback",只是把"找不到"和"加载失败"做了区分。

### `keys() -> list`

返回所有**可用**的 agent 名——包括 `_skill_map` 里发现的真实 skill 名,以及 `_legacy_mapping` 里指向存在 skill 的别名。**不触发加载**。

### `values()` / `items()`

**只返回已缓存的实例**。这是有意的设计:没加载的 Agent 没有实例可返回。如果想知道"所有可用 agent",用 `keys()`。

### `get_loaded_agents() -> list`

`list(self.cache.keys())`,用于 CLI 的 `/status` 命令展示"当前已加载的 Agent"。

## 典型用法

**在 CLI 初始化阶段注册**:

```python
from agents.lazy_agent_registry import LazyAgentRegistry

agent_cache = {}
registry = LazyAgentRegistry(
    model=shared_model,
    cache=agent_cache,
    memory_manager=memory_manager,
)

# 此时没有任何 skill 被加载,仅知道它们存在
print(registry.keys())  # ['research', 'marketing', 'product', ...]
```

**OrchestrationAgent 按优先级调度**:

```python
for task in plan.tasks:
    agent = registry[task.intent]   # 首次访问触发加载
    result = await agent.reply(msg)
```

**CLI 状态命令**:

```python
loaded = registry.get_loaded_agents()
available = registry.keys()
console.print(f"已加载 {len(loaded)}/{len(available)} 个 skill")
```

## 设计要点

- **发现 vs 加载分离**: 启动成本恒定(只扫目录),真实成本随用户使用路径摊销。对 CLI 这种冷启动敏感的场景尤其重要——用户问一个"研究市场"的问题,只会加载 ResearchAgent,marketing/product/sales/web/monitoring 全部躺在磁盘上。
- **`sys.modules` 注册**: `module_from_spec` 创建的模块必须写入 `sys.modules[module_name]`,否则同一模块内的 `from . import ...` 相对导入会失败,且多次加载会产生不同类对象(`isinstance` 出诡异 bug)。
- **`sys.path` 副作用**: 每次加载都会尝试把项目根目录插入 `sys.path`(有去重)。这是为了让 skill 目录内的 `from utils import ...` 能工作——skill 不是 Python 包的一部分,走的是脚本加载路径,必须手动补 path。
- **`inspect.getmembers` 找类**: 约定 skill 的 `agent.py` 必须有**至少一个** `AgentBase` 的子类,按文件内出现顺序取第一个。如果一个文件里定义了多个 Agent 类,只会拿到其一——这是潜在坑点,目前项目约定一个 skill 一个 Agent 类。
- **构造参数按签名注入**: `inspect.signature` 检查 `__init__` 有没有 `memory_manager`,这样 skill 开发者可以选择是否依赖记忆系统。但 `model` 和 `name` 是强契约,所有 Agent 必须接受。
- **缓存是外部传入**: 上层可以通过清空 `agent_cache` 实现"热重载"所有 Agent(改了 agent.py 后下次访问会重新 `exec_module`),也可以只清单个 key 做针对性重载。当然,`sys.modules` 里的旧模块不会被清,严格的热重载需要配合 `importlib.reload` 或 `del sys.modules[...]`。
- **不是线程安全的**: `cache` 是普通 dict,`__getitem__` 里的 check-then-set 没有锁。多线程并发首次加载同一个 agent 会竞态(可能实例化两次)。CEOClaw 单事件循环 async 不受影响;如果放到多线程服务里要包 `Lock`。
- **加载失败的可见性**: 任何导入、类发现、实例化的异常都会打红色日志 + 完整 traceback 再 raise。配合 CLI 的错误提示,用户能快速定位是哪个 skill 坏了,而其他 skill 继续可用——这正是插件系统的价值。
