# 意图识别智能体 (IntentionAgent)

**源文件**: `agents/intention_agent.py`

## 概述

IntentionAgent 是 CEO Agent 的意图识别层，负责将用户的自然语言输入转化为结构化的调度决策。它通过单次 LLM 调用完成四个步骤：推理过程生成、多意图识别、Query 改写、构建结构化决策（agent 调度计划）。输出的 JSON 直接传给 OrchestrationAgent 执行。

## 在系统中的位置

```
用户输入
  → cli.py 构建 context_messages (List[Msg])
  → IntentionAgent.reply(context_messages)
  → 输出结构化 JSON (Msg)
  → OrchestrationAgent.reply(intention_result)
```

## 依赖组件

- `SkillLoader`（`utils/skill_loader.py`）：动态加载 `.claude/skills/*/SKILL.md` 的元数据，生成可调度 agent 的描述文本注入 prompt。
- `extract_json_from_async_response`（`utils/json_parser.py`）：从 LLM 异步响应中提取文本。
- `robust_json_parse`（`utils/json_parser.py`）：鲁棒的 JSON 解析，支持 6 层容错策略。

## 类: `IntentionAgent(AgentBase)`

### `__init__(self, name: str = "IntentionRecognitionAgent", model=None, **kwargs)`

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `name` | `str` | `"IntentionRecognitionAgent"` | Agent 名称 |
| `model` | 可调用对象 | `None` | LLM 模型实例 |
| `**kwargs` | - | - | 未使用，兼容 AgentBase |

**内部状态**:
- `self.name`, `self.model`
- `self.conversation_history`: `List[str]` — 每次 `reply()` 调用时重新构建，不跨调用保留
- `self.skill_loader`: `SkillLoader` 实例，负责读取 SKILL.md 元数据

---

### `async reply(self, x: Optional[Union[Msg, List[Msg]]] = None) -> Msg`

意图识别主流程。接收用户查询（含上下文），返回结构化的调度决策。

**输入**:

| 参数 | 类型 | 说明 |
|---|---|---|
| `x` | `None` | 返回空 JSON 的 Msg |
| `x` | `Msg` | 单条消息，取 `x.content` 作为用户查询，无上下文 |
| `x` | `List[Msg]` | 消息列表，最后一条为用户查询，其余为上下文 |

**实际调用方式**（`cli.py:170-184`）:

CLI 构建 `context_messages` 列表传入，结构为：

```python
[
    Msg(role="system", content=long_term_summary),   # 可选，长期记忆摘要
    Msg(role="user", content="上一轮用户输入"),         # 短期记忆中的历史消息
    Msg(role="assistant", content="上一轮助手回复"),
    ...
    Msg(role="user", content=user_input)              # 当前用户输入（最后一条）
]
```

**输出**: `Msg(name=self.name, content=json_string, role="assistant")`

其中 `content` 是 JSON 字符串，结构如下：

```json
{
    "reasoning": "推理过程文本，说明如何理解用户 query、如何结合上下文、如何识别意图信号",

    "intents": [
        {
            "type": "research",
            "confidence": 0.95,
            "description": "该意图的具体说明",
            "reason": "为什么识别出该意图"
        }
    ],

    "key_entities": {
        "industry": "行业领域",
        "product_type": "产品类型",
        "target_market": "目标市场",
        "action": "具体操作",
        "other": "其他关键信息"
    },

    "rewritten_query": "标准化、补全后的查询内容",

    "agent_schedule": [
        {
            "agent_name": "research",
            "priority": 1,
            "reason": "调用该智能体的原因",
            "expected_output": "期望输出"
        },
        {
            "agent_name": "product",
            "priority": 2,
            "reason": "需要 research 的结果",
            "expected_output": "产品规划方案"
        }
    ]
}
```

**内部逻辑（按顺序）**:

#### 1. 解析输入，提取用户查询和上下文

- 若 `x` 为 `None`：直接返回空 JSON 的 Msg。
- 若 `x` 为 `List[Msg]`：
  - 取最后一条的 `content` 作为 `user_query`。
  - 遍历前面的消息，按 `role` 分类：
    - `role="system"` → 格式化为 `[系统记忆]\n{content}`
    - `role="user"` → `用户: {content}`（超过 800 字符截断加 `...`）
    - 其他 role → `助手: {content}`（同样截断）
  - 将格式化后的字符串存入 `self.conversation_history`。
- 若 `x` 为单个 `Msg`：取 `x.content` 作为 `user_query`，无上下文。

#### 2. 构建上下文字符串

将 `self.conversation_history` 分为两类：
- 以 `[系统记忆]` 开头的条目 → `system_memory`（取最后一条）
- 其余 → `dialogue_history`

按 `system_memory` 在前、`dialogue_history` 在后的顺序用 `\n` 拼接为 `context_str`。若无上下文则为 `"无历史对话"`。

#### 3. 动态加载 Skills 描述

通过 `SkillLoader.get_skill_prompt()` 读取所有 `.claude/skills/*/SKILL.md` 的 frontmatter，生成可调度 agent 的描述文本。使用固定的 `skill_mapping` 字典（6 个 skill 名称一一映射到自身）。

#### 4. 构建 Prompt 并调用 LLM

Prompt 包含以下部分：
- 当前时间（年月日 + 时分 + 星期）
- 用户 Query
- 对话历史上下文
- 可调度的子智能体列表（来自 SkillLoader）
- 意图区分原则（语义理解，非关键词匹配）
- 四步任务要求（推理 → 多意图识别 → Query 改写 → 结构化决策）
- 输出格式要求（严格 JSON）
- 优先级设置规则（P1 信息收集 → P2 策略生成 → P3 执行部署）

LLM 调用参数：
```python
messages = [
    {"role": "system", "content": "你是一个高级意图识别专家。只输出JSON格式的结果，不要输出其他文本。"},
    {"role": "user", "content": prompt}
]
response = await self.model(messages)
```

#### 5. 解析 LLM 响应

1. `extract_json_from_async_response(response)` — 从异步响应（可能是流式生成器）中提取文本。
2. `robust_json_parse(text, fallback=default_result)` — 鲁棒解析 JSON，支持 6 层容错：
   - 直接解析
   - 移除控制字符
   - 修复单引号→双引号
   - 移除尾部逗号
   - 转义字符串内换行符
   - json5 宽松解析（若可用）

#### 6. 兜底策略

若 LLM 调用抛出异常，使用 `default_result`：

```python
{
    "reasoning": "意图识别出错，使用默认策略。",
    "intents": [{"type": "research", "confidence": 0.5, ...}],
    "key_entities": {},
    "rewritten_query": user_query,  # 保留用户原话
    "agent_schedule": [{"agent_name": "research", "priority": 1, ...}]
}
```

默认调度 `research` agent，置信度 0.5。

## 优先级规则

Prompt 中定义了三个优先级层级，同优先级的 agent 并行执行：

| 优先级 | 类型 | 包含的 Agent | 说明 |
|---|---|---|---|
| Priority 1 | 信息收集 | research, monitoring, sales | 互不依赖，可并行 |
| Priority 2 | 策略生成 | marketing, product | 依赖 P1 的结果 |
| Priority 3 | 执行部署 | web | 依赖 P1 + P2 的结果 |

**示例**: 用户说"帮我调研AI教育市场，然后做产品方案和落地页"
- P1: research（调研）
- P2: product（产品规划，依赖调研结果）
- P3: web（落地页，依赖产品方案）

## 设计要点

- **单次 LLM 调用**: 推理、意图识别、Query 改写、调度决策在一个 prompt 里完成，不做多轮。
- **显示推理（Chain-of-Thought）**: prompt 要求 LLM 先输出 `reasoning` 字段再给出决策，利用 CoT 提升意图识别准确度。
- **动态 Skill 发现**: 通过 SkillLoader 读取 SKILL.md，新增 skill 不需要修改 IntentionAgent 代码。但 `skill_mapping` 字典和 prompt 中的优先级规则是硬编码的，新增 skill 后需要手动更新。
- **上下文截断**: 历史消息超过 800 字符会被截断，防止 prompt 过长。
- **conversation_history 无状态**: 每次 `reply()` 调用都重新构建 `self.conversation_history`，不会累积历史。
- **兜底为 research**: LLM 失败时默认调度 research agent，这是最安全的选择（只做信息收集，不产生副作用）。
