# 调度智能体 (OrchestrationAgent)

**源文件**: `agents/orchestration_agent.py`

## 概述

OrchestrationAgent 是 CEO Agent 的**协调调度层**,负责接收 IntentionAgent 的调度决策,按优先级编排和执行多个 skill agent,并聚合结果返回给用户。它是整条多 agent 流水线的"大脑":决定谁先执行、谁能并行、前序结果如何传递、异常如何降级,以及哪些产出要写入长期记忆。

## 在系统中的位置

```
用户输入
  → IntentionAgent.reply()          # 产出 agent_schedule(含priority)
  → OrchestrationAgent.reply()      # 本文件
    ├─ _prepare_context()           # 从 MemoryManager 补齐对话历史/偏好
    ├─ 按 priority 分批:
    │   ├─ P1 parallel: research / monitoring / sales
    │   ├─ P2 parallel: marketing / product
    │   └─ P3 sequential: web
    ├─ _aggregate_results()         # 聚合为统一结构
    └─ _update_memory()             # 选择性写入长期记忆
  → 返回 Msg 给 cli.py 格式化展示
```

OrchestrationAgent **不直接调用 LLM**,它是纯控制流 agent——所有 LLM 推理都委托给被调度的 skill agent。

## 依赖组件

- **`agent_registry`** (`Dict[str, AgentBase]`):由 `LazyAgentRegistry` 注入的 skill agent 实例表,key 是 SKILL.md 的 `name` 字段(research/marketing/product/sales/monitoring/web)
- **`MemoryManager`**(`context/memory_manager.py`):可选。提供 `short_term.get_recent_context(n)` 和 `long_term.get_preference()` / `save_project_history()`
- **`asyncio.gather`**:同优先级 agent 的并行执行引擎,`return_exceptions=True` 保证单个 agent 失败不影响整批

### `async reply(self, x: Optional[Union[Msg, List[Msg]]] = None) -> Msg`

调度主流程。接收 IntentionAgent 的输出,按优先级执行 skill agent,聚合结果。

**输入**:

| 参数 | 类型 | 说明 |
|---|---|---|
| `x` | `None` | 返回 `{"error": "No input provided"}` |
| `x` | `Msg` | 取 `x.content`,需为 IntentionAgent 的 JSON 输出 |
| `x` | `List[Msg]` | 取最后一条的 `content` |

**期望的输入内容**(IntentionAgent 的产出):

```json
{
    "reasoning": "...",
    "intents": ["research", "product"],
    "key_entities": {...},
    "rewritten_query": "为AI教育产品做调研并规划功能",
    "agent_schedule": [
        {
            "agent_name": "research",
            "priority": 1,
            "reason": "需要市场调研",
            "expected_output": "调研报告"
        },
        {
            "agent_name": "product",
            "priority": 2,
            "reason": "基于调研做产品规划",
            "expected_output": "产品规划"
        }
    ]
}
```

**输出**: `Msg(name=self.name, content=json_string, role="assistant")`

聚合后的典型结构:
```json
{
    "status": "completed",                    // 或 "partial_failure" / "no_agents"
    "intention": {
        "intents": [...],
        "key_entities": {...}
    },
    "agents_executed": 2,
    "results": [
        {
            "agent_name": "research",
            "priority": 1,
            "status": "success",
            "data": {"findings": "...", "results": {...}}
        },
        {
            "agent_name": "product",
            "priority": 2,
            "status": "success",
            "data": {"product_plan": {...}}
        }
    ],
    "errors": 0                               // 仅 partial_failure 时出现
}
```

**内部逻辑**:

#### 1. 解析输入

将 `Msg.content`(或 `List[Msg]` 的最后一条)按 JSON 解析。解析失败直接返回 `{"error": "Invalid intention format"}`——这是硬失败,不尝试修复。

#### 2. 取 `agent_schedule` 并按 priority 排序

```python
sorted_schedule = sorted(agent_schedule, key=lambda x: x.get("priority", 999))
```

缺失 priority 的任务会被排到最后(999)。空 schedule 返回 `{"status": "no_agents"}`。

#### 3. 准备 context

调用 `_prepare_context(intention_data)` 构建传给 skill agent 的上下文:

```python
context = {
    "reasoning": "...",         # 来自 IntentionAgent
    "intents": [...],
    "key_entities": {...},
    "rewritten_query": "...",
    "recent_dialogue": [...],   # 来自 short_term.get_recent_context(3)
    "user_preferences": {...}   # 来自 long_term.get_preference()
}
```

没有 `memory_manager` 时,`recent_dialogue` 和 `user_preferences` 字段不会出现。

#### 4. 按 priority 分批并行执行

核心调度循环。遍历已排序任务,当 priority 发生变化时,把累积的同优先级任务一次性发给 `_execute_parallel_agents`:

```python
current_priority = None
parallel_tasks = []
for task in sorted_schedule:
    priority = task.get("priority", 0)
    if current_priority is not None and priority != current_priority:
        # 执行上一批
        batch_results = await self._execute_parallel_agents(parallel_tasks, context, results)
        results.extend(batch_results)
        parallel_tasks = []
    current_priority = priority
    parallel_tasks.append(task)
# 收尾执行最后一批
```

**关键性质**:
- **同 priority 并行,跨 priority 顺序**:P1 全部完成后才启动 P2,P2 全部完成后才启动 P3。
- **`previous_results` 累积**:每批启动时,`results`(当前已完成的全部批次)都会作为 `previous_results` 传入,后批 agent 能看到前批产出。
- **单任务优化**:`_execute_parallel_agents` 内部,`len(tasks) == 1` 时直接 await,不走 `asyncio.gather`。

#### 5. 聚合与记忆写入

调用 `_aggregate_results()` 得到统一结构,然后(若有 memory_manager)调用 `_update_memory()` 选择性写入长期记忆。

---

### `_prepare_context(self, intention_data) -> Dict`

见"主流程第 3 步"。内存查询是**同步**调用,会阻塞事件循环;当前实现假设 MemoryManager 的查询足够快。

---

### `async _execute_parallel_agents(self, tasks, context, previous_results) -> List[Dict]`

批量执行同优先级 agent。

**逻辑**:
- 空任务列表 → 返回 `[]`
- 单任务 → 直接 `await self._execute_agent(...)`,避免 `gather` 开销
- 多任务 → 为每个任务创建协程,`asyncio.gather(*coros, return_exceptions=True)` 并行执行

**异常隔离**: `return_exceptions=True` 使某个 agent 抛异常不会中断整批。异常会被包装成:
```json
{
    "status": "error",
    "agent_name": "...",
    "data": {"error": "<异常消息>"},
    "message": "并行执行失败: ..."
}
```

---

### `async _execute_agent(self, agent_name, context, reason, expected_output, previous_results) -> Dict`

执行单个 skill agent。**所有 skill agent 的输入消息都在这里构建**,这也是前几份文档里反复出现的那份 `input_msg` 结构:

```python
input_msg = Msg(
    name="Orchestrator",
    content=json.dumps({
        "context": context,
        "reason": reason,
        "expected_output": expected_output,
        "previous_results": previous_results
    }, ensure_ascii=False),
    role="user"
)
response = await agent.reply(input_msg)
```

**响应处理**:
1. `response.content` 若是字符串,尝试 `json.loads`;失败则包成 `{"output": ...}`
2. 若解析后 dict 有 `error` 字段 → 返回 `status="error"`
3. 否则返回 `status="success"`,`data` 为 agent 返回的字典

**异常捕获**: 整个 try 块兜底。`agent_name` 未注册 → 直接返回 error(不抛异常),确保调度层稳定。

---

### `_aggregate_results(self, results, intention_data) -> Dict`

组装最终返回结构。见"输出"段。

**status 决策**:
- 默认 `"completed"`
- 任一 agent 的 `result.status == "error"` → `"partial_failure"` + `errors` 计数
- 上游已过滤了 `no_agents` 分支,不会到这里

注意:**不存在全失败的 `"failed"` 状态**——哪怕所有 agent 都报错,这里仍然是 `"partial_failure"`。

---

### `_update_memory(self, intention_data, results)`

把 skill agent 的产出选择性写入 `long_term.project_history`。

**写入规则**(硬编码):

| agent_name | 读取字段(OR 取其一) | 写入的 action_type |
|---|---|---|
| `product` | `strategy` / `product_plan` | `"product_planning"` |
| `marketing` | `strategy` / `marketing_plan` | `"marketing_strategy"` |
| `research` | `findings` / `research_result` | `"research"` |

**关键点**:
- `description` 字段取字符串化产物的前 200 字,完整 `data` 另存
- `monitoring`、`sales`、`web` 的结果**不写入长期记忆**——只有"形成决策"的 agent 才持久化
- 此方法是同步的,但在 `reply()` 里以同步方式调用(不 await),因此若 MemoryManager 内部有 I/O,会阻塞事件循环

## 完整调用链示例

用户输入"为AI教育产品做调研并规划功能"后的完整流程:

```
1. IntentionAgent 产出:
   agent_schedule = [
     {agent_name: "research", priority: 1, ...},
     {agent_name: "product",  priority: 2, ...}
   ]

2. OrchestrationAgent.reply()
   → 解析 JSON
   → 按 priority 排序: [research(P1), product(P2)]
   → _prepare_context() 合并 reasoning + 短期对话 + 长期偏好

3. 第一批 (P1): [research]
   → len==1, 单任务路径
   → _execute_agent("research", context, reason, expected_output, previous_results=[])
   → 构建 input_msg 调用 ResearchAgent.reply()
   → 返回 {status:"success", data:{findings, results}}
   → results = [{agent_name:"research", priority:1, result:{...}}]

4. 第二批 (P2): [product]
   → _execute_agent("product", context, reason, expected_output, previous_results=results)
   → ProductAgent 在 prompt 的 all_info 中看到 research 的产出
   → 返回 {status:"success", data:{product_plan:{...}}}

5. _aggregate_results() 生成 {status:"completed", agents_executed:2, results:[...]}

6. _update_memory():
   → research: 写入 long_term (action_type=research)
   → product:  写入 long_term (action_type=product_planning)

7. 返回给 cli.py,Rich 格式化展示
```

## 调度模型特性对比

| 维度 | 本调度器 | 常见替代方案 |
|---|---|---|
| 调度粒度 | priority 分层 + 层内并行 | DAG 显式依赖 |
| 依赖传递 | `previous_results` 累积全部前批 | 按 edge 传递特定输出 |
| 并行控制 | `asyncio.gather(..., return_exceptions=True)` | Task pool / Semaphore 限流 |
| 异常策略 | 单 agent 失败不中断批次,整体 `partial_failure` | Fail-fast / Retry / Circuit breaker |
| LLM 调用 | **零次**(纯控制流) | LLM-as-router 再决策 |
| 记忆写入 | 硬编码三类 agent | 由 agent 自行决定 |

---

## 设计要点

- **纯控制流,不耗 token**: OrchestrationAgent 自己不调 LLM,所有推理都发生在 skill agent 内部。这意味着调度本身是确定性的、可预测的、零成本的。
- **priority 既是分批信号也是并行边界**: 同值并行、不同值顺序,这个简单规则取代了显式 DAG。代价是无法表达"research 只喂给 product,不喂给 marketing"这种细粒度依赖——**所有 P2 agent 都能看到所有 P1 产出**。
- **`previous_results` 单调累积**: 后批 agent 收到的 `previous_results` 是**迄今为止所有已完成批次**的结果,不只是紧邻的上一批。P3 的 WebAgent 因此能同时看到 P1 和 P2 的产出。
- **异常强隔离**: 两层 try/except(`_execute_agent` 内 + `gather(return_exceptions=True)`)确保调度循环不会因单 agent 崩溃而断。结果里永远有所有被调度 agent 的条目,只是 status 可能是 error。
- **记忆写入是白名单**: 只有 research/product/marketing 的产出会进长期记忆,且要求字典里有特定键(`strategy`/`product_plan`/`findings`/…)。不符合就悄悄跳过——这里没有日志提示字段缺失,是个容易被忽略的静默失败点。
- **没有重试和熔断**: 本文件不含 `CircuitBreaker` 或重试逻辑;那些在 skill agent 内部或 `utils/llm_resilience.py` 里实现。调度层只做"调用一次,成功/失败如实上报"。
- **Memory 查询是同步阻塞**: `_prepare_context` 和 `_update_memory` 都以同步方式调用 MemoryManager,若 MemoryManager 将来引入磁盘 I/O 或远程存储,需要改成 async 以免阻塞并行批次的事件循环。
- **JSON 硬失败**: 若 IntentionAgent 输出不是合法 JSON,`reply()` 直接返回 error,不做容错修复。这是合理的——意图识别层已有 6 层 JSON 容错 + LLM 修复兜底,到调度层就应该是干净数据。
